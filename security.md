## Container vulnerability scanning

Image scans may report vulnerabilities originating from the base image (Debian slim).

These are typically classified as:

- OS library vulnerabilities
- No upstream fix available

The container is hardened to reduce exploitability.

Base images are periodically updated when upstream fixes are released.

# Threat Model (STRIDE)

This project uses the **STRIDE threat model** to analyze potential risks.

| Type | Meaning |
|------|--------|
| S | Spoofing (identity impersonation) |
| T | Tampering (manipulation of data) |
| R | Repudiation (denial of actions) |
| I | Information Disclosure (data leakage) |
| D | Denial of Service |
| E | Elevation of Privilege |

# STRIDE Analysis

## Spoofing

### Threat

An attacker may attempt to provide a manipulated CIS benchmark document.

### Risk

very-Low

------------------------------------------------------------------------

## Tampering

### Threat

Manipulation of:

-   input benchmark files
-   generated CSV/JSONL output
-   container image

### Risk

Low

### Mitigation

The parser records multiple integrity fields:

-   `source_pdf_sha256`
-   `block_text_sha256`
-   `parser_version`
-   `extracted_at_utc`

This allows traceability at all times:

PDF → control → CSV record

Additionally, results are sorted:

benchmark_name → benchmark_version → control_id

This enables diffing and integrity verification.

------------------------------------------------------------------------

## Repudiation

### Threat

A user denies that a file was generated by the parser.

### Mitigation

The output contains metadata:

-   benchmark_name
-   benchmark_version
-   benchmark_date
-   source_pdf_sha256
-   parser_version
-   extracted_at_utc

This allows verification of the data origin.

------------------------------------------------------------------------

## Information Disclosure

### Threat

Unintentional leakage of sensitive information via logs or output.

### Risk

Low

### Reason

The tool processes only **public CIS benchmark documents**.

No:

-   credentials
-   API keys
-   external services

are used.

The tool operates fully offline.

------------------------------------------------------------------------

## Denial of Service

### Threat

Malicious PDF files may attempt to:

-   crash parser libraries
-   cause excessive memory consumption

### Risk

Medium

### Mitigation

Container sandboxing is recommended:

docker run --read-only --tmpfs /tmp --cap-drop ALL --security-opt no-new-privileges

This limits the impact of a potential parser vulnerability.

------------------------------------------------------------------------

## Elevation of Privilege

### Threat

An exploit in the PDF parser may attempt to escalate privileges.

### Risk

Low

### Mitigation

The container runs:

-   as a **non-root user**
-   without Linux capabilities
-   with **no-new-privileges**
-   with a read-only filesystem

Example secure runtime:

docker run --read-only --tmpfs /tmp --cap-drop ALL --security-opt no-new-privileges cis-pdf2csv
